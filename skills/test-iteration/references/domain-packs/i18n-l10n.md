# Pack — i18n / l10n

Pull when the change touches translated strings, locale formatting, RTL, or text encoding. Internationalization = the code can handle any locale; localization = a specific locale is correct.

## Text & encoding
- [ ] UTF-8 end-to-end: full character range reads/writes/displays; data written in one language reads correctly under another
- [ ] No hard-coded user-facing strings; all externalized to resource files
- [ ] No string **concatenation** to assemble messages (breaks grammar/order across locales)
- [ ] Font fallback works — no tofu (□) for missing glyphs

## Locale formatting
- [ ] Date / time / number / currency format **per locale** (e.g. `DD/MM/YYYY` vs `MM/DD/YYYY`, decimal & thousands separators, currency symbol placement, negative-number convention)
- [ ] Calculations stay correct after localized formatting (parse back correctly)
- [ ] Time-zone conversion correct (and per-tenant where applicable); logs/audit timestamps consistent
- [ ] Collation / sort / comparison / search return locale-appropriate order

## Layout & RTL
- [ ] RTL locales (Arabic/Hebrew): UI mirrored, navigation/icons/charts flow correctly, bidi (mixed LTR/RTL) renders
- [ ] Text expansion/contraction: longer translations don't overflow, overlap, or truncate
- [ ] Pluralization and grammar correct in translated validation/error messages

## Behaviour
- [ ] Missing translation → correct **fallback** language, not a blank or a key
- [ ] Selected locale persists across logout / tenant switch (no reset to default)

> Sources: Microsoft Globalization — Internationalization Checklist & How-to-perform i18n testing; globalization-testing guides (Brigita, MasterSoftwareTesting).
