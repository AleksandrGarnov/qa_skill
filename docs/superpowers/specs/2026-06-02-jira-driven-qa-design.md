# Дизайн: Jira-driven QA для плагина `qa-skill`

- **Дата:** 2026-06-02
- **Статус:** утверждён к реализации
- **Источник идей:** паттерны из `tac-4` (AI Developer Workflows), отфильтрованные под задачу тестирования + сверка с индустриальными best practices (агентный QA, Jira-driven test design).

## Проблема

`test-iteration` сейчас выводит, что тестировать, **только из git diff + `CLAUDE.md`**. Связи с тем, что задача/баг вообще должны были делать (acceptance criteria, шаги воспроизведения), нет. Это признанный анти-паттерн: «tests written from implementation details, not requirement statements». Зелёный CI доказывает, что код работает изолированно, но не отвечает «фича полностью покрыта?». Кроме того, exit criteria сейчас — проза, интерпретируемая на глаз, поэтому вердикт GO/NO-GO невоспроизводим.

## Цель (scope этого захода)

Три связанных улучшения (п.1–3 из проработки):

1. **Jira как точка входа QA** — тянуть суть задачи/бага + acceptance criteria + шаги воспроизведения из тикета и трассировать к ним тесты и вердикт. Дифф остаётся вторичным источником.
2. **Детерминированный exit-gate** — exit criteria из прозы → измеримые бинарные критерии (`метрика ≥ порог`) + исполняемые validation-команды, где это возможно; вердикт считается fail-closed.
3. **Матрица трассировки** в отчёте — `AC → тесты → статус → дефекты` + orphan-detection в обе стороны.

### Вне scope (отдельным заходом)

Классификатор QA-стратегии по типу тикета, машиночитаемый баг-формат, suspension/resumption гейты, run-ID логи. Никакого Python/Pydantic-слоя, авто-PR, записи обратно в Jira — **только чтение тикета**. `.env`/секреты не читаем.

## Архитектура

Сохраняем текущую философию плагина: много маленьких независимо вызываемых скиллов; `test-iteration` композирует их. Новое подключается ровно так же, как уже подключены `branch-review`/`qa-research`.

### Новое

- `skills/jira-context/SKILL.md` — независимо вызываемый скилл: `ключ тикета → Jira → структурированный контекст (AC/repro)`.
- `scripts/jira-key.sh <branch>` — детерминированное извлечение ключа тикета (read-only, ничего не пишет).

### Правим (композиция, не переписывание)

- `skills/test-iteration/SKILL.md` — шаги 1, 5, 6, 9.
- `skills/test-iteration/references/test-report-template.md` — добавить матрицу трассировки.
- `skills/test-iteration/references/manual-checklist-template.md` — колонка `trace → AC`.
- `.claude-plugin/plugin.json` — bump версии.

## Компонент: `jira-context`

Поток из трёх шагов:

1. **Resolve key** (`jira-key.sh`): regex `[A-Z][A-Z]+-\d+` (case-sensitive, официальный формат Jira). Приоритет источников: **имя ветки → заголовок PR (`gh`, если доступен) → не-merge коммиты**. Несколько совпадений → канон берётся из имени ветки. Ничего не найдено → возвращает `NONE` (не угадывает).
2. **Fetch** через Atlassian MCP `getJiraIssue`. Минимальный набор полей (остальное — шум): `key, issuetype, summary, description, priority, components, labels, attachments (только наличие), issuelinks, status` + acceptance criteria. AC в этом проекте лежат **внутри `description`** — это основной путь: парсить описание на блоки AC (Given/When/Then, чеклисты, нумерованные пункты). Мягкий фолбэк для совместимости/эволюции проекта: кастомное AC-поле (определять через `getJiraIssueTypeMetaWithFields`) → Checklist-приложение. **Фолбэк по доступу:** MCP недоступен или ключ = `NONE` → попросить пользователя вставить текст тикета.
3. **Extract** → структурированный блок: `summary, issuetype, AC[], (для бага: S2R / expected / actual), priority, components, issuelinks`.
   - AC отсутствуют/неполны → **явный флаг** «AC missing/inferred» (тесты из головы не пишем — это сигнал, а не повод).
   - AC неявные (в стене текста) → реконструировать в Given/When/Then **декларативно** (поведение, не UI-детали) + добавить edge/negative кейсы, пометив «выведено, требует подтверждения PO».

## Компонент: детерминированный exit-gate

В `test-iteration` шаг 6 exit criteria переводятся из прозы в **измеримые бинарные** критерии (`метрика ≥ порог`, напр. «critical-path pass rate ≥ 99%», «open blocker == 0»). Где исполнимо — формируется список validation-команд (по образцу секции «Validation Commands» из tac-4), каждая обязана завершиться без ошибок. Шаг 9: вердикт считается **fail-closed** против зафиксированных на шаге 6–7 критериев — не свежее суждение на момент отчёта. Деферралы кодируются как явные owned-waiver'ы (severity + mitigation + owner + дата), а не молчаливый downgrade.

## Компонент: матрица трассировки

В `test-report-template.md` добавляется матрица:

| AC / требование | Пункты чек-листа | Статус (pass/fail/blocked) | Связанные дефекты |
|---|---|---|---|

Плюс **orphan-detection** в обе стороны: AC без покрывающих тестов (forward — дыры в покрытии) и тесты без основания в AC (backward — лишнее/угаданное). Это аудит-артефакт каждого пре-мёрджа.

## Интеграция в цикл `test-iteration`

- **Шаг 1 (контекст):** дополнительно вызвать `jira-context`; AC/repro вливаются в проектный контекст наряду с `CLAUDE.md`.
- **Шаг 5 (triage):** каждый AC становится конкретным «что проверить»; git diff — вторичный источник (что реально менялось).
- **Шаг 6:** пункты чек-листа трассируются к ID критериев/AC; exit criteria измеримы и бинарны.
- **Шаг 9:** в отчёт добавляется матрица трассировки; вердикт fail-closed против зафиксированных критериев.

## Безопасность / tool-agnostic

- Atlassian MCP **опционален** и деградирует в «вставь текст тикета»; никакого no-op без предупреждения.
- Ключ тикета — детерминированный скрипт, не угадывание модели.
- **Нет** хардкод-URL, кредов, ключей проектов, имён кастомных полей; проект сообщает специфику сам.
- `.env`/секреты не читаются; если шагу нужна переменная — спрашиваем пользователя.
- Скрипт `jira-key.sh` строго read-only против git/`gh`; ничего не пушит и не пишет в Jira.

## План поставки — две PR-ветки (одна фаза = один PR)

1. **PR 1 — фича** (ветка `feat/jira-driven-qa`): скилл `jira-context`, скрипт `jira-key.sh`, правки `test-iteration` (шаги 1/5/6/9), обновлённые reference-шаблоны, bump версии в `plugin.json`. Сюда же — этот спек.
2. **PR 2 — документация** (отдельная ветка): полноценное описание новой возможности в проектной доке (`README.md` + при необходимости отдельный раздел/страница в `docs/`), чтобы у плагина было цельное, законченное описание Jira-driven QA. Отдельный PR — чтобы ревьюер мог принять фичу и доку независимо.

## Критерии готовности

- `jira-context` достаёт ключ детерминированно и тянет минимальный набор полей через MCP, с фолбэком на ручную вставку.
- Отсутствие/неполнота AC явно флагается, а не маскируется.
- Exit criteria в `test-iteration` измеримы и бинарны; вердикт fail-closed.
- Отчёт содержит матрицу трассировки с orphan-detection в обе стороны.
- Никаких хардкод-секретов/URL; `.env` не читается; всё деградирует на bare-install.
